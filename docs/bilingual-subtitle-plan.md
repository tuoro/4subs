# 双语字幕架构说明

## 当前目标

`4subs` 现在的目标很明确：

- 输入：本地影片资源
- 处理：尽量把重计算交给远程 API
- 输出：可人工校对的双语 `SRT` / `ASS`

## 当前处理链路

系统已按下面顺序执行：

1. 扫描本地媒体目录
2. 创建任务并进入后台队列
3. 优先提取外挂字幕或内嵌文本字幕轨
4. 找不到文本字幕时，抽帧并调用远程 OCR 识别硬字幕
5. OCR 仍失败时，提取音频并调用远程 ASR 转写
6. 将字幕整理成标准 `SRT` 时间轴块
7. 调用 `DeepSeek API` 批量翻译
8. 输出双语 `SRT` / `ASS`
9. 在前端详情页预览、编辑、保存成品字幕

## 为什么翻译后时间轴不会乱

核心原则是：

- 时间轴来自源字幕 / OCR 恢复结果 / ASR 转写结果
- 翻译阶段只替换文本内容
- 不会重新计算字幕块的 `Start` / `End`

也就是说，翻译并不会把字幕重新打散，而是附着在已有时间轴块上。

## 各模块职责

### `internal/jobrunner`

负责任务调度、状态推进、取消、回退和失败处理。

### `internal/media`

负责：

- 查找外挂字幕
- 提取内嵌文本字幕
- 提取 ASR 音频
- 截取 OCR 关键帧

### `internal/ocr`

负责：

- 远程视觉 OCR 适配
- OCR 结果整理
- OCR 文本时间轴恢复

### `internal/asr/openai`

负责远程语音转写，并返回带时间轴的字幕块。

### `internal/translator/deepseek`

负责批量翻译字幕内容，并尽量保持顺序稳定。

### `web`

负责：

- 媒体扫描
- 任务创建
- 运行状态查看
- 日志查看
- 字幕预览与人工校对

## 当前边界

当前版本已支持：

- 文本字幕提取
- OCR 硬字幕识别
- ASR 音频转写兜底
- DeepSeek 翻译
- 双语 `SRT` / `ASS` 输出
- 在线预览与编辑
- 任务取消与重试
- GitHub Actions 构建 Docker 镜像

当前仍值得继续增强：

- OCR 结果缓存
- OCR 失败重试
- OCR 帧去重与成本控制
- 术语表导入导出
- 字幕版本管理

## 最符合当前诉求的部署思路

因为你的目标是“本地轻、远程重”，所以推荐配置是：

- 翻译：`DeepSeek API`
- OCR：远程视觉模型 API
- ASR：远程音频转写 API

这样本地主要只承担：

- 文件扫描
- `ffmpeg` 抽帧
- `ffmpeg` 提音频
- 成品字幕写出

这正是当前项目已经落地的架构方向。
